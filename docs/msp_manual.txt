==========================================================
MSP 운영팀 운영 매뉴얼 v1.0
==========================================================

----------------------------------------------------------
1. 팀 개요
----------------------------------------------------------
MSP 운영팀은 고객사 데이터센터 내 VM(가상머신) 및 Kubernetes(K8s) 환경에 배포된 솔루션을 모니터링하고 운영하는 역할을 담당한다.
현재 주요 관리 솔루션은 Upstage Solar Pro이며, 폐쇄망 환경에서 운영된다.

----------------------------------------------------------
2. 모니터링 기본 절차
----------------------------------------------------------
2-1. 일상 모니터링 주기
- 실시간: Grafana 대시보드를 통해 CPU, 메모리, 디스크 I/O 상태 확인
- 매 1시간: Pod 상태 및 로그 이상 여부 점검
- 매일 오전 9시: 전일 장애 이력 및 알람 발생 현황 리뷰
- 매주 월요일: 리소스 사용 트렌드 분석 및 증설 필요 여부 검토

2-2. 핵심 모니터링 지표
- CPU 사용률: 경고 70%, 위험 90%
- 메모리 사용률: 경고 75%, 위험 90%
- 디스크 사용률: 경고 80%, 위험 95%
- Pod 재시작 횟수: 1시간 내 3회 이상 시 장애로 간주
- API 응답 시간: 정상 2초 이내, 경고 5초 초과, 위험 10초 초과

----------------------------------------------------------
3. VM 운영 절차
----------------------------------------------------------
3-1. VM 상태 확인
- vSphere Client 또는 SSH를 통해 VM 상태 점검
- 명령어: vmware-cmd --server [HOST] --username [USER] getstate [VM_PATH]
- VM이 응답 없을 경우 해당 고객사 담당자에게 즉시 연락

3-2. VM 재시작 절차
1) 고객사 사전 승인 획득 (Jira 티켓 생성 필수)
2) 서비스 영향도 확인 (연결된 서비스 목록 체크)
3) 재시작 전 스냅샷 생성
4) 재시작 실행 후 5분 이내 정상 기동 확인
5) 서비스 정상 여부 확인 후 티켓 완료 처리

3-3. VM 디스크 증설
- 증설 요청은 고객사 담당자 승인 후 진행
- 증설 가능 최대 용량: 단일 VM 기준 2TB
- 증설 후 파티션 확장 명령어: resize2fs /dev/sda1

----------------------------------------------------------
4. Kubernetes(K8s) 운영 절차
----------------------------------------------------------
4-1. 클러스터 상태 확인
kubectl get nodes                          # 노드 상태 확인
kubectl get pods -n [NAMESPACE] -o wide    # Pod 상태 확인
kubectl describe pod [POD_NAME] -n [NAMESPACE]  # Pod 상세 정보
kubectl logs [POD_NAME] -n [NAMESPACE] --tail=100  # 로그 확인

4-2. Pod 재시작
kubectl rollout restart deployment/[DEPLOYMENT_NAME] -n [NAMESPACE]

4-3. Pod CrashLoopBackOff 대응
1) 로그 확인: kubectl logs [POD_NAME] -n [NAMESPACE] --previous
2) 이벤트 확인: kubectl describe pod [POD_NAME] -n [NAMESPACE]
3) OOM(메모리 부족)인 경우: 리소스 limit 상향 후 재배포
4) 이미지 오류인 경우: 이전 버전으로 롤백
   kubectl rollout undo deployment/[DEPLOYMENT_NAME] -n [NAMESPACE]

4-4. 노드 NotReady 대응
1) 노드 상태 확인: kubectl describe node [NODE_NAME]
2) 노드 SSH 접속 후 kubelet 상태 확인: systemctl status kubelet
3) kubelet 재시작: systemctl restart kubelet
4) 10분 내 복구 안 될 시 고객사 에스컬레이션

----------------------------------------------------------
5. Solar Pro 운영 절차
----------------------------------------------------------
5-1. Solar Pro 개요
- Upstage에서 개발한 LLM 솔루션
- 고객사 폐쇄망 내 K8s 환경에 배포되어 운영
- 네임스페이스: solar-pro
- 주요 컴포넌트: solar-api-server, solar-model-server, solar-gateway

5-2. Solar Pro 상태 확인
kubectl get pods -n solar-pro
kubectl logs -n solar-pro deployment/solar-api-server --tail=50

5-3. Solar Pro API 응답 불가 시 대응
1) API 서버 Pod 상태 확인
   kubectl get pods -n solar-pro
2) 모델 서버 메모리 사용량 확인 (GPU 메모리 OOM 여부)
   kubectl top pods -n solar-pro
3) API 서버 재시작
   kubectl rollout restart deployment/solar-api-server -n solar-pro
4) 재시작 후 5분 대기 → 헬스체크 API 호출로 정상 여부 확인
   curl http://[SOLAR_ENDPOINT]/health
5) 복구 안 될 시 Upstage 기술지원팀 연락 (support@upstage.ai)

5-4. Solar Pro 모델 서버 GPU OOM 대응
1) GPU 메모리 사용량 확인: nvidia-smi
2) 불필요한 프로세스 종료 후 모델 서버 재시작
3) 반복 발생 시 배치 크기(batch size) 축소 설정 변경 요청

5-5. Solar Pro 업데이트 절차
1) Upstage로부터 신규 이미지 태그 수령
2) 스테이징 환경에 먼저 배포 후 검증 (최소 30분)
3) 고객사 유지보수 시간 협의 (보통 새벽 2시~4시)
4) 프로덕션 배포: kubectl set image deployment/solar-api-server solar-api-server=[NEW_IMAGE]
5) 배포 후 롤아웃 상태 확인: kubectl rollout status deployment/solar-api-server -n solar-pro

----------------------------------------------------------
6. 장애 대응 절차
----------------------------------------------------------
6-1. 장애 등급 분류
- P1 (Critical): 서비스 전체 중단. 즉시 대응, 30분 내 고객사 보고
- P2 (High): 주요 기능 장애. 1시간 내 대응
- P3 (Medium): 일부 기능 저하. 4시간 내 대응
- P4 (Low): 경미한 이슈. 다음 영업일 내 대응

6-2. 장애 발생 시 공통 절차
1) 장애 인지 즉시 Jira 티켓 생성 (등급 명시)
2) 팀 Slack 채널 #msp-incident에 상황 공유
3) 고객사 담당자 연락 (P1, P2는 전화 필수)
4) 원인 파악 및 조치
5) 서비스 복구 확인
6) 장애 보고서 작성 (복구 후 24시간 이내)

6-3. 에스컬레이션 기준
- 30분 내 원인 파악 불가 시: 팀장 에스컬레이션
- 2시간 내 복구 불가 시: 고객사 CTO 보고
- Solar Pro 자체 버그로 판단 시: Upstage 기술지원팀 연락

----------------------------------------------------------
7. 접근 권한 및 보안
----------------------------------------------------------
7-1. 접근 권한 원칙
- 최소 권한 원칙 적용: 업무에 필요한 최소 권한만 부여
- 고객사 시스템 접근 시 VPN 필수 사용
- 개인 계정으로 접근 금지, 팀 공용 계정 사용

7-2. 접근 기록
- 모든 SSH 접근은 Bastion 서버를 통해 이루어지며 자동 기록
- kubectl 명령어 실행 이력은 audit log에 자동 저장
- 분기별 접근 권한 검토 실시

7-3. 보안 사고 발생 시
1) 즉시 해당 계정 비활성화
2) 팀장 및 보안팀 보고
3) 접근 로그 보존 (덮어쓰기 금지)
